# roughlogic.com Specification v592 -- Live Answer Preview and Search UX Polish (app.js, index.html, data/search/preview-map.json, 0 New Tiles)

> **Status: LANDED (2026-07-10, package 0.179.0). Platform spec, last of the "ask it a question" search series (v589-v592). Depends on
> v589 (ranker, viaTypo) and v591 (slot mapping).** No new tile, module, or dependency; every behavior below is a
> deterministic function of the query and bundled data -- the answer preview calls the same exported compute functions
> the tiles call. Inherits spec.md through spec-v591.md.
>
> **As-landed deltas.** (1) The preview-map entry carries two fields the sketch in §1 omitted and the feature cannot work
> without: `args` (DOM input id -> compute **argument name** -- computes take named parameters, not DOM ids) and
> `defaults` (the renderer's example-fill values for arguments the slots do not supply, without which most computes
> return non-finite and no preview would ever render). (2) The did-you-mean row fires when the **top** match needed the
> typo pass, not when *every* match did -- a generic token like "fill" always matches some tile exactly, so the
> every-match condition never fires, including on this spec's own "conduit fill" example. (3) The no-match fallback's
> group-coverage rows are unreachable by construction (rankTools returns every tool with coverage > 0, so zero results
> implies zero group signal); the landed fallback is the "Browse all 21 trades" row. (4) rankTools rows expose
> `typoFixes` (corrected corpus token per query token) to feed the did-you-mean copy. (5) Home-payload js grew ~2.6 KB
> gz (35,934 B, 71.6% of the sub-budget) vs the <= 2 KB estimate; no cap change.
>
> **The gap, and the evidence for it.** After v591, *"voltage drop 120v 150 ft 20 amps"* routes to a prefilled tile --
> but the user still has to click through to learn the answer. The compute functions are pure, exported, and lazy-loadable
> per module, so the answer can render **inside the search dropdown** the moment the slots map. That is the product's
> strongest differentiator stated in one line: ask a question, get the computed, cited answer, offline, with no AI
> involved at all. Three smaller dead-ends ship alongside: the no-match row is a hard stop (no path to the group hubs),
> a typo-corrected result set never says so, and the placeholder copy teaches tool names, not questions, under a label
> that literally asks "What do you need to figure out?".

## 1. Live answer preview

**The map.** A new lazy shard `data/search/preview-map.json` (strict JSON; sw.js precache + manifest row), generated by a
build script from `test/fixtures/compute-map.js` plus a hand-authored headline table, covering exactly the v591
slot-seeded tiles:

```
{ "voltage-drop": { "module": "./calc-electrical.js", "fn": "<export>",
                    "headline": [{ "key": "<result key>", "label": "drop", "unit": "V", "decimals": 1 }] } }
```

**The behavior.** When the **top-ranked** result has a slot row with >= 1 mapped quantity and a preview-map entry: after
a 150 ms debounce, lazily `import()` the module (calc modules are already lazy-per-tile; this is the same cost the click
would pay a moment later), call the compute with the mapped params over the tile's defaults, and -- if the result is
finite and not `{ error }` -- append a `span.sr-preview` to that dropdown row: `voltage drop 3.6 V`. Any failure
(import, throw, error object, missing headline key, partial inputs the compute rejects) renders nothing; the preview can
only add, never block or replace, the normal result row. The option's accessible name includes the preview text via the
existing combobox semantics (`aria-activedescendant` reads the full option), so no extra live region is added. Values
format with `Number(v).toFixed(decimals)` -- the site's no-thousands-separator output convention.

**The gate.** `scripts/check-slots.mjs` (v591's lint) extends to the preview map: every preview tile has a v591 slot
row; every `module`/`fn` import-resolves; and each headline `key` is asserted present by running the tile's
worked-example inputs through the function (the worked-example runner technique) -- a renamed output key fails CI, not
the user.

## 2. Search UX polish (all deterministic)

**Did-you-mean.** When the rendered result set exists only via the v589 typo pass (`viaTypo` on every match), prepend a
non-option presentation row (like `.search-empty`): `showing matches for "conduit fill"` -- the corrected token string,
so the user learns the vocabulary instead of wondering why the results look right.

**No-match fallback.** On zero ranked results, replace the dead-end copy with up to 3 rows linking the trade-group hubs
whose tiles carried the best partial token coverage (plain links to the prerendered `/groups/<slug>/` shells), plus a
final "Browse all 21 trades" row anchoring the home browse-by-trade index. A dead end becomes a fork in the road.

**Placeholder rotation.** `index.html` keeps its shipped placeholder as the static fallback; on boot, app.js swaps in
one entry from a fixed ~8-item list of example *questions* (*"how many yards of concrete for a 10x12 slab"*, *"what size
wire for 50 amps at 120 ft"*, ...), selected by day-of-month modulo list length -- deterministic per day, no timers, no
animation, nothing for prefers-reduced-motion to object to. The `.hero-label` is unchanged; the placeholder is never the
accessible name.

## 5. Roadmap position

Closes the v589-v592 series. With it, the honest end-to-end story is: strip the question frame (v589), know the
vocabulary (v590), take the numbers (v591), show the answer (v592) -- deterministic at every step, offline, cited on the
tile the user lands on. Deliberately out of scope, recorded here for the paper trail: any runtime language model,
embedding search, or classifier (evaluated 2026-07-10 and rejected -- payload, job-site phone performance, and the
README "No AI" table row all argue against; routing accuracy of small on-device models loses to this deterministic
stack for a 942-tile near-neighbor catalog). If semantic-recall evidence ever accumulates, that debate reopens in a new
spec with an explicit opt-in design; nothing in v589-v592 forecloses it.
