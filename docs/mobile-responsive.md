# Mobile-Responsive Sweep (spec-v12 Phase F)

This document is the per-tile mobile-responsive sweep checklist from spec-v12 §10 (Phase F). It carries forward as the running record of the sweep state across releases. The sweep target is **zero horizontal scroll at any viewport width >= 320 px** on the home view and on every tile view.

## 1. The reported bug (F.1, fixed 2026-05-12)

The `.v6-reference-block` in [styles.css](../styles.css) used `grid-template-columns: max-content 1fr` on the dt+dd list with `white-space: nowrap` on dt. On a 375 px viewport the long dt label "Edition selector / disclosure" plus a long dd value (a codes.iccsafe.org URL, an NFPA TOC URL, or a CFR section path) exceeded the viewport. The grid did not collapse because `max-content` does not shrink below intrinsic min-width.

**Fix landed in commit f57ca6e** (spec-v12 draft + Phase F.1 mobile reference-block fix). The `@media (max-width: 760px)` breakpoint now overrides `grid-template-columns: 1fr` and `white-space: normal` on the dt and dd, and adds `overflow-wrap: anywhere` and `word-break: break-word` on the dd. Manual visual checks at 320 / 375 / 414 / 760 px viewport widths confirmed the reference block now wraps cleanly under all four widths.

## 2. Per-tile sweep invariants (F.2)

At each release window, every new tile passes these invariants at 320 px (iPhone SE 1st gen), 375 px (modern iPhone SE / 12 mini), 414 px (iPhone Plus), and 760 px (the existing breakpoint):

- **No horizontal scroll** on the home view at any width >= 320 px.
- **No horizontal scroll** on any tile view at any width >= 320 px.
- **48 px touch target floor** on every interactive element (already enforced by [styles.css](../styles.css) per spec.md §11; the sweep verifies it on every new tile).
- **Long strings wrap cleanly.** URLs in the citation footer, multi-clause source-stamp lines, hyphenated tile names, and the limitation-banner replacement copy all get `overflow-wrap: anywhere`. No mid-word overflow.
- **`inputmode` on numeric inputs.** Every numeric field uses `inputmode="decimal"` / `numeric` / `tel` so the soft keyboard surfaces the right pad. The [ui-fields.js](../ui-fields.js) `makeNumber` helper handles this for every v12 tile by default; spot-check on any tile that opts out.
- **Print view paginates.** The existing v5 print-table CSS handles multi-page output cleanly; the sweep verifies print at A4 and Letter on every tile with a long table (amortization, NIHSS, anesthesia vitals, peds vitals, FMR areas, etc.).

## 5. Voice-input round-trip (F.4)

Voice input on iOS Safari and Android Chrome dispatches an `input` event (not a `change` event) on the input field. Every v12 numeric input uses `addEventListener("input", ...)` in its renderer, so a voice-input value triggers compute without requiring a subsequent keystroke. The `compute on input` invariant is asserted indirectly by the v10 §C runner (every fixture compute is exercised) and explicitly by the v9 §F.2 timer-jitter test in [test/unit/calc-field-v9.test.js](../test/unit/calc-field-v9.test.js).

## 8. Wide-table screen overflow (F.6, fixed in the v0.14 window)

A second class of horizontal-scroll bug, distinct from the F.1 reference-block grid, lived in the multi-column data tables. The `.tabular-tool` wrapper (loan amortization, mileage roll-up, PCR master mix, and the other schedule-style tiles in [calc-accounting.js](../calc-accounting.js), [calc-lab.js](../calc-lab.js), [calc-historical.js](../calc-historical.js), and [calc-plumbing.js](../calc-plumbing.js)) only carried `@media print` rules. On screen the wrapper was a plain block with `overflow: visible`, so a five-column currency table (loan amortization: Month / Payment / Principal / Interest / Balance, each cell a `$1,234.56`-width value) has an intrinsic width of roughly 350-420 px and pushed the entire `.view-region` wider than a 320 px phone, producing a page-level horizontal scrollbar.

**Fix.** [styles.css](../styles.css) now gives `.tabular-tool` `overflow-x: auto` (plus `-webkit-overflow-scrolling: touch`) on screen, so the wrapper owns the horizontal scroll and the page never does. The table stays fully readable with a contained swipe; the print block continues to render the table full-width on paper. A companion `.out-value` rule adds `min-width: 0; overflow-wrap: anywhere` so a long unbreakable result token (a binary base-conversion string, a long hex stability pin) wraps inside its flex output line instead of overflowing.

**Regression guard.** [test/integration/a11y.test.js](../test/integration/a11y.test.js) asserts `documentElement.scrollWidth <= clientWidth + 1` at a 320 px viewport on the home view, on `#loan-amortization` and `#macrs-depreciation` (the widest data tables, populated via their example buttons), on `#color-codes` (the longest reference `<dl>`), and on `#rent-vs-buy` (the finance tile with the longest single-line output strings). The prerendered `/tools/<id>/` and `/groups/<slug>/` shells are covered separately by the §10 shell audit.

## 9. Wide-table follow-up: a table that escaped the `.tabular-tool` class (fixed 2026-06-08)

The F.6 fix (§8) gave `.tabular-tool` `overflow-x: auto` on screen, but that only protects tables whose wrapping `<div>` actually carries the class. A 2026-06-08 full-catalog audit -- a one-shot Playwright sweep that routes to all 437 tile ids at 320 px, clicks the example button, and asserts no page-level horizontal scroll -- found exactly one offender: **`macrs-depreciation`** (445-450 px wide vs the 320 px floor). Its renderer in [calc-accounting.js](../calc-accounting.js) built the five-column schedule (Year / Percent / Depreciation / Accumulated / Book value) inside a bare `<div id="mc-out-schedule">` that was never given the `.tabular-tool` class, so the F.6 CSS never applied to it and the intrinsic table width pushed the page wide. The sibling tables in the same file (loan amortization, the other schedule-style outputs) all carry the class; the MACRS `<div>` was the lone miss.

**Fix.** Added `tableWrap.className = "tabular-tool"` to the MACRS renderer so the wrapper owns the horizontal scroll like every other schedule table. After the fix the full-catalog audit reports **437 / 437 tiles clean at 320 px** (**555 / 555** at the current catalog size).

**The audit is now a standing gate (2026-06-08).** The one-shot harness was promoted to a permanent CI check: the `responsive-stress.test.js` "every live tile view: no page-level horizontal scroll at 320 px" case (§11) loads the SPA once, walks every TOOLS id via `location.hash` (the app re-renders on `hashchange`), populates each tile's example output, and asserts no page-level scroll -- the full-catalog sweep, on every push. Hash-navigation rather than a per-tile reload keeps the whole 555-tile sweep to ~30-45 s, fast and non-flaky enough for the integration job, so the curated §8 spot-check is now a fast first-failure signal layered over full catalog coverage rather than the only coverage. A new tile that ships a schedule `<div>` without the `.tabular-tool` class -- the exact `macrs-depreciation` regression -- now fails the build instead of escaping.

**Takeaway for new schedule tiles.** Any renderer that emits a multi-column `<table>` MUST place it inside a `<div class="tabular-tool">`. The class -- not the table -- is what owns the scroll; a table in a bare `<div>` re-introduces the F.6 bug one tile at a time. The contributor checklist in §7 item 1 is satisfied for table tiles only when the wrapper carries `.tabular-tool`.

## 10. Prerendered-shell mobile guard (added 2026-06-09)

The §8 / §9 guards cover the interactive SPA tile views (the `#hash` routes). The prerendered static shells under `dist/tools/<id>/`, `dist/groups/<slug>/`, and `changelog.html` are a **separate, zero-JS surface** -- the pages crawlers and direct-link visitors land on -- and the always-on integration job serves the repo root (not `dist/`), so it never reaches them. They had been verified only by manual spot-check.

[scripts/check-shell-mobile.mjs](../scripts/check-shell-mobile.mjs) (`npm run check:shell-mobile`, after `npm run build`) closes that gap: it serves `dist/` and asserts `documentElement.scrollWidth <= clientWidth + 1` at 320 px on **every** prerendered shell -- 555 tool shells + 24 group hubs + the changelog + the home shell, 581 in all. The run reports **581 / 581 clean**. The shells are structurally lower-risk than the SPA tiles (a shell renders an `<h1>`, a description, a JSON-LD block, a citation, a worked example, and a related-tiles list -- no input form, no schedule table), so the only overflow vector is a long unbreakable string, which the shared `overflow-wrap: anywhere` rule already handles.

The audit is an on-demand gate rather than part of `npm run lint`, because it needs a headless browser (Playwright is a CI-only no-save dependency) and a built `dist/`; it skips cleanly (exit 0, with a message) when either is absent, mirroring the dist-absent skip in `check-module-sizes.mjs`. Run it after any change to the shell template ([scripts/build-shells.mjs](../scripts/build-shells.mjs)) or to [styles.css](../styles.css). The Lighthouse CI job already builds `dist/` and audits a sample of these shells for performance and accessibility; this guard adds the horizontal-scroll dimension across the full shell set.

## 11. Beyond a single viewport: text-zoom + landscape + tablet gate (added 2026-06-09)

§8-§10 all assert no horizontal scroll at a 320 px **viewport**. That single axis also satisfies WCAG 2.2 **1.4.10 Reflow** -- 400% page zoom on a 1280 px window resolves to a 320 CSS-px viewport, which is exactly what the sweep tests. But two other responsive axes go untested by a viewport-only sweep, and a 2026-06-09 audit confirmed the fluid layout (`clamp()` type, `ch` / `vw` measures, `max-width` column caps, one `760 px` breakpoint) holds on both. Each is guarded over the surface the harness can actually reach, because the two surfaces are served differently:

- **WCAG 1.4.4 Resize Text (200% text-only zoom).** Firefox and the OS text-zoom affordance scale font size without scaling the viewport, so a fixed-width or short-word container can spill past its column even though the page never scrolled at 320 px. Both gates emulate it by doubling the root `font-size` (so every `rem`-based size, control height, and padding grows) and re-asserting `scrollWidth <= clientWidth + 1`.
- **Landscape phones + tablet portraits.** 568x320 and 667x375 (short-height landscape), 768 and 834 (iPad / iPad Air portrait), and both sides of the lone 760 px breakpoint (759 / 761).

**SPA surface** -- [test/integration/responsive-stress.test.js](../test/integration/responsive-stress.test.js) drives the interactive views on three axes: the **full catalog** (all 555 live tile views, each with its example output populated) at the 320 px portrait floor (§9), nine diverse SPA routes (home, a phasor-SVG calculator, both wide-table schedules, the longest reference `<dl>`, a select-heavy tile, and the longest-output finance tiles) at 200% text zoom, and a four-route subset across the six landscape/tablet viewports. The integration webServer serves the **repo root**, where the prerendered `/tools/` and `/groups/` shells do **not** exist (they live in `dist/`), so the test asserts a 2xx on every navigation before measuring -- otherwise a 404 page, which trivially does not scroll, would be a false pass. (An earlier revision of this test listed shell routes directly and was silently measuring 404s; that is exactly the trap the 2xx guard now closes, and shell coverage moved to the auditor below.)

**Prerendered-shell surface** -- [scripts/check-shell-mobile.mjs](../scripts/check-shell-mobile.mjs) serves `dist/` and so is the only harness that can reach the shells. It audits **every** shell (581) at the 320 px floor, and -- because the shells are generated from one template -- a representative sample (an evenly-strided slice of tool shells plus every group hub, the changelog, and the home shell) at landscape + 200% text zoom. It waits for network-idle only on the client-rendered changelog (the static shells are painted at the load event), which keeps the whole 677-check sweep under ~9 s.

The SPA gate runs inside the existing `test:e2e` job (Playwright `testMatch` picks up any `*.test.js`); the shell auditor runs in the same integration job after `npm run build`. Together they are the regression lock for the "every view scrolls only vertically, on every device class, at any readable text size" guarantee.

**Two-engine coverage (added 2026-06-09).** The no-horizontal-scroll guarantee is the one responsive axis worth verifying on a second browser engine: WebKit (the iOS Safari engine) resolves flexbox `min-width`, intrinsic table width, and sub-pixel rounding differently from Chromium, and roughly half of US mobile traffic is iOS Safari. [playwright.config.js](../test/integration/playwright.config.js) defines two projects -- the default `chromium` runs the whole suite (smoke, layout, a11y/axe, print, CSV, render-leak, perf, and `responsive-stress`), and a scoped `webkit-responsive` project (`testMatch: /responsive-stress\.test\.js$/`) re-runs *only* the responsive sweep on WebKit, so all three axes (320 px portrait full-catalog, 200% text-zoom, landscape/tablet) are checked on both engines while the rest of the integration job stays Chromium-only and bounded. The CI integration job installs both browsers (`npx playwright install --with-deps chromium webkit`). A 2026-06-09 full-catalog WebKit run reported **555 / 555 tiles clean** at the 320 px floor.
